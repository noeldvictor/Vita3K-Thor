## What The Screenshot Shows

- The visible screenshot is not Vita3K. ADB reports the foreground app as `net.rpcsx.easy/net.rpcsx.MainActivity`.
- Vita3K was only in the background as `org.vita3k.emulator.debug/org.vita3k.emulator.Emulator`.
- The screenshot shows the RPCSX frontend: title bar `RPCSX`, wide cropped game tiles, large floating `Cheats`/`Patched` badges, a home tile mixed into the game grid, and a large empty lower half.
- That specific visual issue should be treated as an RPCSX/frontend layout problem unless reproduced inside Vita3K.
- RPCSX logcat also shows it selected a custom driver named `Turnip A8XX Draft-v1` on this AYN Thor. Thor is Adreno 740/a7xx, so an a8xx-focused driver is suspicious for graphics testing even if the current screenshot is mostly UI layout.

## Vita3K Back/Select Check

- Launched `org.vita3k.emulator.debug/org.vita3k.emulator.Emulator` with ADB.
- Captured the Vita3K start screen before/after ADB `KEYCODE_BACK` and `KEYCODE_BUTTON_SELECT`.
- The screen did not change and no Runtime OSD opened, which matches current code because the Runtime OSD opens only when a game/title is running.
- ADB synthetic Select produced SDL log lines: `Unknown keycode 109`.
- The real Thor controller is more complex than ADB keyevent injection. `getevent -lp` shows the `Odin Controller` exposes `KEY_BACK`, `KEY_APPSELECT`, `BTN_SELECT`, `BTN_START`, shoulders, sticks, and d-pad.
- This can explain weird physical behavior: Back/Select may arrive through key events, gamepad button events, or both depending on Android/SDL mapping.

## OSD Input Fix Direction

- Add a temporary runtime input diagnostic mode that logs SDL event type, key scancode/keycode, gamepad button, gamepad instance ID, and whether the event opened OSD or consumed a chord.
- Reproduce with a running Vita title, not the Vita3K start screen.
- Treat short Back/Select as OSD only when a game is running.
- Keep `Select + R1`, `Select + right-stick down`, and `Select + right-stick up` chord-first so short-press release does not also open/close the OSD.
- Consider accepting `KEY_APPSELECT` and `BTN_SELECT` as Select only after physical logs prove which path Thor emits.

## Graphics Debugging/Profiling Plan

- First split issues into frontend/UI, host renderer, and guest Vita rendering. The captured RPCSX grid issue is frontend/UI, not a Vita GXM rendering bug.
- For Vita3K in-game graphics bugs, capture per-title reports with:
  - title ID and game/update version
  - renderer/backend and selected Turnip/system driver
  - resolution multiplier and GPU settings
  - screenshot/video
  - logcat tail
  - GXM/display call trace around the bad frame
  - shader translation/pipeline state for the bad draw
  - surface/texture cache metadata
- Add optional debug toggles to dump current frame, display frame info, active render target, shader hash, pipeline state, and texture uploads.
- Use Android profiling only after the bug is reproducible: `dumpsys SurfaceFlinger`, `dumpsys gfxinfo`, Perfetto/simpleperf, and renderer timing counters.

## Ghidra And Vita Architecture Notes

- Ghidra can help when a legally dumped game behaves differently from emulator assumptions. Use it to identify which Vita imports the title uses, which SceGxm paths it exercises, and whether it relies on timing/display behavior we stub or approximate.
- Do not commit commercial game binaries, decrypted modules, license material, or proprietary databases.
- Use Vita-aware loading. VitaLoaderRedux is a Ghidra extension for Vita ELF-PRX modules and can mark module entry points, imports, exports, and NIDs.
- Relevant architecture facts for emulator debugging:
  - Sony's PS Vita product outline lists a 4-core ARM Cortex-A9 CPU and SGX543MP4+ GPU.
  - VitaSDK documents SceGxm as the Vita GPU graphics library.
  - SceGxm initialization includes a display queue callback used for display swaps, which is a key path for render/display bugs.
  - VitaSDK notes VPK packaging around SELF executables and mentions crash core dumps in `ux0:data` for homebrew debugging.
